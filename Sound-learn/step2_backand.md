# Step 2 Backend 마일스톤 — Piano Roll 시각화 지원

**작성일:** 2026-04-11
**대상 로드맵:** Step 2 (3~4주차)
**담당 영역:** Backend (FastAPI / Python)
**문서 목적:** 현재 구현 상태를 기준으로 Step 2 완료를 위해 백엔드에서 마무리해야 할 작업을 마일스톤 방식으로 정리

---

## 1. Step 2 목표 해석

README 기준 Step 2의 목표는 다음과 같다.

- 음정 데이터를 시각적으로 표현
- Canvas 기반 그래프 구현
- MIDI 기준 멜로디 렌더링
- 사용자 Pitch overlay

현재 프론트엔드에는 Piano Roll UI가 이미 존재하므로, 백엔드 관점의 핵심 목표는 아래 두 가지다.

- 프론트가 안정적으로 시각화할 수 있는 pitch 데이터를 일관된 형식으로 제공
- 이후 MIDI 기준 멜로디 overlay를 붙일 수 있도록 응답 구조와 데이터 계약을 정리

---

## 2. 현재 백엔드 구현 상태

### 완료된 항목

- FastAPI 서버 구동 가능
- `/api/upload` 업로드 엔드포인트 구현
- WAV / MP3 / WebM 업로드 검증 구현
- 22050Hz 모노 리샘플링 구현
- `librosa.pyin` 기반 pitch 추출 구현
- 무성 구간 `null` 처리 구현
- pitch 요약 통계 반환 구현
- 로컬 프론트 연동용 CORS 설정 반영

### Step 2 기준 부족한 항목

- 업로드 응답 스키마 문서화 부족
- Step 2 완료 기준에 맞는 백엔드 산출물 정의 부족
- 프론트 시각화 기준의 데이터 품질 검증 부족
- MIDI 기준 멜로디 데이터 API 부재
- 예외/경계 테스트 문서화 부족

---

## 3. Step 2 완료 조건

아래 조건을 만족하면 백엔드 기준 Step 2 완료로 본다.

- 프론트가 `/api/upload` 응답만으로 사용자 pitch를 안정적으로 시각화할 수 있다
- pitch frame 데이터 형식과 null 처리 규칙이 문서화되어 있다
- 시각화에 필요한 요약값이 일관되게 반환된다
- 대표 예외 케이스에 대한 테스트 결과가 있다
- MIDI 기준 멜로디 overlay를 위한 다음 단계 인터페이스 초안이 준비되어 있다

## 4. 마일스톤 구성

## Milestone 1. API 응답 계약 고정

### 목표

프론트가 의존하는 업로드 응답 형식을 Step 2 기준으로 확정한다.

### 해야 할 일

- [ ] `/api/upload` 성공 응답 필드 확정
- [ ] `pitch` 배열 구조 문서화
- [ ] `summary` 객체 구조 문서화
- [ ] 실패 응답 형식 문서화
- [ ] 파일 확장자 / 용량 제한 / 상태 코드 표 정리

### 산출물

- 백엔드 API 명세 문서
- 샘플 성공 응답 JSON
- 샘플 실패 응답 JSON

### 완료 기준

- 프론트엔드 담당자가 별도 문의 없이 응답 형식을 이해할 수 있음

---

## Milestone 2. Pitch 데이터 품질 검증

### 목표

Piano Roll 시각화에 필요한 최소 품질을 만족하는 pitch 데이터를 보장한다.

### 해야 할 일

- [ ] 프레임 간격(`time`)이 일정한지 확인
- [ ] `frequency` 반올림 규칙 점검
- [ ] `midi_note` 반올림 규칙 점검
- [ ] 무성 구간 `null` 처리 기준 확인
- [ ] 이상치 점프 제거 로직 확인
- [ ] 짧은 녹음 / 긴 녹음 각각에서 결과 확인

### 점검 포인트

- 시각화 시 포인트가 과도하게 튀지 않는지
- 무성 구간이 잘못 유성으로 표시되지 않는지
- 프레임 수가 지나치게 많아 프론트 렌더링을 방해하지 않는지

### 산출물

- 테스트용 샘플 오디오별 결과 JSON
- 품질 점검 체크리스트

### 완료 기준

- 데모용 업로드 3종 이상에서 시각화 결과가 자연스럽게 보임

---

## Milestone 3. 예외 처리 및 안정성 보강

### 목표

업로드 API가 Step 2 데모 중 쉽게 깨지지 않도록 방어 범위를 확인한다.

### 해야 할 일

- [ ] 지원하지 않는 확장자 업로드 테스트
- [ ] 50MB 초과 파일 업로드 테스트
- [ ] 디코딩 불가 파일 테스트
- [ ] 무성만 포함된 오디오 테스트
- [ ] 임시 파일 생성 후 삭제 여부 확인
- [ ] pitch 추출 실패 시 500 응답 메시지 확인

### 산출물

- 예외 케이스 테스트 결과 표
- 재현 방법 정리

### 완료 기준

- 주요 실패 시나리오에서 서버가 죽지 않고 의도한 응답을 반환함

---

## Milestone 4. 프론트 시각화 연동 점검

### 목표

현재 프론트의 `PianoRoll`과 실제 백엔드 응답이 문제없이 연결되는지 확인한다.

### 해야 할 일

- [ ] 실제 업로드 응답으로 Piano Roll 렌더링 확인
- [ ] duration과 frame 수가 프론트 표시와 맞는지 확인
- [ ] summary 값이 `AnalysisSummary`에서 정상 반영되는지 확인
- [ ] localhost / 127.0.0.1 환경에서 CORS 문제 없는지 확인
- [ ] 녹음 업로드와 파일 업로드 둘 다 동작 확인

### 산출물

- 프론트 연동 확인 기록
- 발견된 인터페이스 이슈 목록

### 완료 기준

- 프론트에서 실제 녹음 혹은 파일 업로드 후 분석 결과가 바로 시각화됨

---

## Milestone 5. MIDI overlay 다음 단계 준비

### 목표

README의 Step 2 요구사항 중 “MIDI 기준 멜로디 렌더링”에 대비해 백엔드 인터페이스 초안을 만든다.

### 해야 할 일

- [ ] 기준 MIDI 데이터를 어떤 형식으로 전달할지 초안 작성
- [ ] 추후 추가할 응답 필드 후보 정의
- [ ] MIDI note event 구조 제안
- [ ] 사용자 pitch와 기준 melody를 함께 반환하는 방식 검토

### 추천 초안 필드

```json
{
  "reference_pitch": [
    { "start": 0.0, "end": 0.5, "midi_note": 60 }
  ],
  "segments": [],
  "meta": {
    "song_id": "demo-song",
    "key": "C",
    "tempo": 120
  }
}
```

### 산출물

- Step 3 확장용 응답 구조 초안

### 완료 기준

- Step 2 종료 시점에 다음 단계 설계를 바로 이어갈 수 있음

## 6. 권장 작업 순서

1. `/api/upload` 응답 스키마를 문서로 고정
2. 실제 오디오 파일로 pitch 품질 확인
3. 예외 케이스 테스트 수행
4. 프론트 `PianoRoll` 실연동 검증
5. MIDI overlay용 차기 응답 구조 초안 작성
