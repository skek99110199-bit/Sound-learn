# Step 2 변경점 요약

**작성일:** 2026-04-12
**기준 비교:** Step 1 대비 Step 2에서 기술적으로 달라진 점 요약
**목적:** Step 1 문서를 읽은 사람이 Step 2에서 무엇이 실제로 확장되었는지 빠르게 파악할 수 있도록 정리

---

## 1. 한 줄 요약

Step 1이 **“업로드와 pitch 추출이 되는 상태”**였다면, Step 2는 **“추출된 pitch 결과를 프론트에서 실제 Piano Roll과 요약 정보로 시각화하고, 실사용 업로드 흐름까지 검증한 상태”**까지 확장되었다.

---

## 2. Step 1과 Step 2의 핵심 차이

| 구분 | Step 1 | Step 2 |
|------|--------|--------|
| 백엔드 응답 | 업로드/분석 기반 최소 응답 | `pitch` + `summary` 구조를 프론트 시각화 기준으로 확정 |
| pitch 데이터 활용 | 추출 가능 수준 | 실제 그래프 렌더링에 사용 |
| 프론트 결과 화면 | 녹음/업로드 중심 | 분석 결과 화면 제공 |
| 시각화 | 없음 또는 데모 수준 | `PianoRoll` 실제 렌더링 |
| 요약 정보 | 미정 또는 제한적 | 길이, 감지 프레임, 음역 범위 표시 |
| 사용자 검증 | 업로드 성공 여부 중심 | 실제 WAV / MP3 / WebM / 녹음 결과까지 확인 |

---

## 3. 백엔드 관점에서 달라진 점

### 3.1 `/api/upload` 응답 계약 확정

Step 1에서는 업로드와 pitch 추출 자체가 중심이었다.

Step 2에서는 아래 응답 구조를 프론트 연동 기준으로 고정했다.

```json
{
  "filename": "recording.wav",
  "duration_sec": 3.3,
  "original_sr": 44100,
  "normalized_sr": 22050,
  "pitch": [],
  "summary": {}
}
```

즉 Step 2부터는 백엔드 응답이 단순 분석 결과가 아니라 **프론트 UI 계약** 역할을 하게 되었다.

---

### 3.2 `pitch` 데이터 규칙 명확화

Step 1에서는 pitch 추출이 된다는 사실이 중요했다.

Step 2에서는 아래 규칙이 명시적으로 정리되었다.

- `time`: 약 23ms 간격
- `frequency`: 소수 둘째 자리 반올림
- `midi_note`: 소수 첫째 자리 반올림
- 무성 구간은 `null`
- 단발 노이즈 프레임 제거
- 과도한 pitch 점프 제거
- 1프레임 무성 구간 보간

즉 Step 2에서는 **시각화 가능한 품질의 pitch 데이터**가 중요해졌다.

---

### 3.3 `summary` 계산 기준 추가

Step 1에서는 결과를 보여주기 위한 요약 정보가 핵심이 아니었다.

Step 2에서는 프론트 결과 카드에 쓰기 위해 아래 요약값 기준을 확정했다.

- `voiced_frames`
- `total_frames`
- `min_frequency`
- `max_frequency`
- `min_midi`
- `max_midi`
- `avg_frequency`

즉 Step 2부터는 백엔드가 단순 분석 엔진이 아니라 **프론트용 요약 통계 제공자** 역할도 하게 되었다.

---

### 3.4 예외 케이스 검증 범위 확대

Step 1은 기능 구현 중심이었다.

Step 2에서는 아래 항목을 실제로 검증했다.

- 잘못된 확장자
- 파일명 없음
- 50MB 초과
- 디코딩 실패
- silence 입력
- temp 파일 정리
- 실제 업로드 흐름에서의 정상 응답 확인

즉 Step 2에서는 **실사용 안정성 확인**이 추가되었다.

---

### 3.5 CORS 및 로컬 실행 조건 보강

Step 1에서는 `localhost:3000` 기준이 중심이었다.

Step 2에서는 실제 로컬 개발 환경 차이를 고려해 아래 origin도 허용했다.

- `http://localhost:3000`
- `http://127.0.0.1:3000`
- `http://localhost:3001`
- `http://127.0.0.1:3001`

즉 Step 2에서는 **실제 개발환경 연결성**이 보강되었다.

---

## 4. 프론트 관점에서 달라진 점

### 4.1 결과 화면이 실제로 사용되기 시작함

Step 1 프론트는 녹음과 업로드가 중심이었다.

Step 2에서는 업로드 후 분석 결과 화면으로 전환되어 아래를 실제로 보여준다.

- 녹음 길이
- 감지된 음 프레임 수
- 음역 범위
- Piano Roll

---

### 4.2 `PianoRoll` 컴포넌트 실사용 검증 완료

Step 2에서 아래 케이스가 실제로 확인되었다.

- 440Hz 테스트 tone: 수평에 가까운 직선
- 실제 녹음: 구간별로 변하는 pitch 분포
- 실제 WAV 파일: 음성 구간이 끊기며 분절된 형태로 표시
- 실제 MP3 파일: 결과 화면 및 그래프 표시 확인
- 실제 WebM 파일: 결과 화면 및 그래프 표시 확인
- silence: 무성 구간 처리

즉 `PianoRoll`은 더 이상 데모가 아니라 **실제 분석 결과 표시 UI**가 되었다.

---

### 4.3 상태 문구 UX 개선

기존에는 녹음 업로드 후 결과 화면 전환이 너무 빨라서 아래 문구가 체감되지 않았다.

- `WAV 변환 중...`
- `분석 중...`
- `분석 완료 (...)`

Step 2에서는 `분석 완료 (...)` 문구를 잠깐 보여주도록 수정했다.

즉 Step 2에서는 **사용자에게 처리 상태가 보이는 UX 개선**도 포함되었다.

---

## 5. 실제 검증 기준에서 달라진 점

Step 1 기준 검증:

- 서버가 뜨는가
- 업로드가 되는가
- pitch가 추출되는가

Step 2 기준 검증:

- 업로드 후 결과 화면이 뜨는가
- `PianoRoll`이 실제 응답으로 렌더링되는가
- `AnalysisSummary`가 숫자를 정상 표시하는가
- 실제 녹음 / WAV / MP3 / WebM 업로드가 되는가
- 상태 문구가 사용자에게 보이는가
- 사용자 입장에서 흐름이 자연스러운가

즉 Step 2는 **기능 존재 여부**에서 **실제 사용자 흐름 검증** 단계로 넘어갔다.
